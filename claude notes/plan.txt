DEFCON OpenGL 3.3 Post-Conversion Bug Fix Plan
=====================================================

## EXECUTIVE SUMMARY
After converting DEFCON from OpenGL 1.2 to OpenGL 3.3 Core Profile, several critical bugs have emerged that break key functionality. This plan addresses these issues by understanding the root causes and implementing targeted fixes.

## ROOT CAUSE ANALYSIS

### 1. SCREEN OPTIONS BLACK SCREEN ISSUE
**Problem**: Changing screen resolution, fullscreen/windowed mode, refresh rate, color depth, z-buffer, or anti-aliasing causes black screen until restart.

**Root Cause**: The `RestartWindowManager()` function in `ScreenOptionsWindow` is empty in the current system:
```cpp
void ScreenOptionsWindow::RestartWindowManager()
{
    // Empty implementation - NO RESTART HAPPENING
};
```

**Original Behavior**: In the original system, this would have triggered `App::ReinitialiseWindow()` which properly recreates the OpenGL context with new settings.

**Current Flow**: 
1. Settings are saved to preferences
2. RestartWindowManager() does nothing
3. Window/context remains with old settings
4. Rendering fails with incompatible state

### 2. SMOOTH LINES & ANTI-ALIASING ISSUES
**Problem**: Graphics menu smooth lines toggle doesn't work, anti-aliasing has no effect.

**Root Cause**: OpenGL 3.3 Core Profile removed the deprecated smoothing functions:
```cpp
// ORIGINAL (OpenGL 1.2):
glEnable(GL_LINE_SMOOTH);
glEnable(GL_POINT_SMOOTH);
glHint(GL_LINE_SMOOTH_HINT, GL_NICEST);

// CURRENT (OpenGL 3.3): 
// These calls were removed but no replacement was implemented
```

**Current Implementation**: The smooth lines code in `BeginScene()` has placeholder comments but no actual implementation.

### 3. LOW DETAIL WORLD NOT WORKING
**Problem**: Low detail world option doesn't change coastline/border rendering detail.

**Root Cause**: The low detail system was designed around OpenGL 1.2 display lists. The conversion to mega-VBO system bypassed the detail level selection:

```cpp
// ORIGINAL: 
if( g_preferences->GetInt(PREFS_GRAPHICS_LOWRESWORLD) == 1 )
{
    // Would select "coastlines-low.dat" 
}

// CURRENT:
if( g_preferences->GetInt(PREFS_GRAPHICS_LOWRESWORLD) == 1 )
{
    // Empty block - no implementation
}
```

### 4. CITY NAMES DISAPPEARING BUG
**Problem**: When city names are disabled, cities disappear completely instead of showing dots without names.

**Root Cause**: City rendering logic was disrupted during batching system implementation. The name rendering and icon rendering are incorrectly coupled.

### 5. PEOPLE TOOLTIP WRONG IMAGES BUG
**Problem**: People tooltip shows upside-down radar.bmp or silo.bmp instead of population density.

**Root Cause**: Texture batching system in OpenGL 3.3 implementation is causing texture state confusion. The `BlitBatched()` system is using incorrect texture binding state:

```cpp
// CURRENT PROBLEMATIC FLOW:
g_renderer->BlitBatched(explosion.bmp) // Population density
// Batching system retains wrong texture state
// When flushed, uses wrong texture (radar.bmp/silo.bmp)
```

# DEFCON Sound System Deep Analysis

Based on comprehensive research of the DEFCON codebase, here's a detailed analysis of how the sound system works across different platforms.

## 🎵 Sound System Architecture Overview

DEFCON uses a sophisticated multi-layered sound system designed for cross-platform compatibility:

### Core Components

1. **SoundSystem** (`soundsystem.cpp/.h`) - Central coordinator
2. **SoundSystemInterface** - Bridge between game world and audio system  
3. **SoundInstance** - Individual sound effect/music instances
4. **SoundLibrary3d** - Abstract base for platform-specific audio backends
5. **SoundSampleBank** - Audio file loading and caching
6. **SoundBlueprintManager** - Configuration-driven sound event system

### Platform-Specific Implementations

## 🎛️ Platform Implementation Details

### Windows Implementation (DirectSound)

**Primary Classes:**
- `SoundLibrary3dDirectSound` - DirectSound 3D wrapper
- Uses hardware-accelerated 3D audio processing
- Supports DSP effects (reverb, chorus, etc.)

**Key Features:**
- **🔧 Sound Library**: DirectSound vs Software 3D processing selection
- Hardware 3D positioning and doppler effects
- Real-time DSP effect chains
- Low-latency audio buffers

**Code Location:** `contrib/systemIV/lib/sound/sound_library_3d_dsound.cpp`

**Initialization Pattern:**
```cpp
#ifdef TARGET_MSVC
g_soundLibrary3d = new SoundLibrary3dDirectSound();
#endif
```

**Channel Management:**
- Uses DirectSound secondary buffers for each channel
- 3D positioning via `IDirectSound3DBuffer` interface
- Real-time frequency/volume adjustments

### Linux/Mac Implementation (SDL + Software 3D)

**Primary Classes:**
- `SoundLibrary2dSDL` - SDL audio output layer
- `SoundLibrary3dSoftware` - Software-based 3D audio processing

**Two-Layer Architecture:**
1. **SDL Layer** (`SoundLibrary2dSDL`): Handles OS audio interface
2. **Software 3D Layer** (`SoundLibrary3dSoftware`): Calculates 3D positioning, mixing

**Code Locations:**
- `contrib/systemIV/lib/sound/sound_library_2d_sdl.cpp`
- `contrib/systemIV/lib/sound/sound_library_3d_software.cpp`

**Initialization Pattern:**
```cpp
#else
g_soundLibrary2d = new SoundLibrary2dSDL();
g_soundLibrary3d = new SoundLibrary3dSoftware();
#endif
```

**Key Features:**
- SDL2 audio callback system
- Software-based stereo mixing with 3D calculations
- Thread-safe buffer management
- Cross-platform OGG/WAV support

## 🎮 Sound Event System

### Event-Driven Architecture

DEFCON uses a blueprint-based system where sounds are triggered by game events:

**Event Types:**
- **Interface Events**: UI feedback (`"Interface", "Error"`)
- **Object Events**: Game object sounds (`SoundObjectId(objectId), "SonarPing"`)  
- **Positional Events**: 3D world sounds (`"Object_nuke", "Detonate", position`)

### Sound Event Triggers (Found throughout codebase)

**Interface Sounds:**
```cpp
// Map interaction
g_soundSystem->TriggerEvent("Interface", "SelectObject");
g_soundSystem->TriggerEvent("Interface", "Error");
g_soundSystem->TriggerEvent("Interface", "HighlightObject");

// Game state changes  
g_soundSystem->TriggerEvent("Interface", "DefconChange");
g_soundSystem->TriggerEvent("Interface", "GameOver");
```

**Object-Attached Sounds:**
```cpp
// Nuclear detonation (nuke.cpp:166)
g_soundSystem->TriggerEvent(SoundObjectId(m_objectId), "Detonate");

// Sonar ping (movingobject.cpp:997)
g_soundSystem->TriggerEvent(SoundObjectId(m_objectId), "SonarPing");
```

**Ambient Systems:**
```cpp
// Background ambience (app.cpp)
g_soundSystem->TriggerEvent("Bunker", "StartAmbience");
g_soundSystem->TriggerEvent("Music", "StartMusic");
```

## 🔊 Audio Processing Pipeline

### 1. Event Processing
1. `TriggerEvent()` called from game code
2. `SoundSystemInterface` resolves object types/positions
3. `SoundBlueprintManager` matches events to sound definitions
4. `SoundInstance` created and queued

### 2. Channel Allocation  
1. `FindBestAvailableChannel()` finds available audio channel
2. Priority system manages channel stealing
3. Music channels vs regular channels separated

### 3. Audio Streaming
1. `SoundLibraryMainCallback()` requests PCM data
2. `SoundSampleHandle` reads from cached audio files
3. Platform-specific mixing and 3D processing
4. Output to OS audio system

### 4. 3D Audio Processing

**Position Types:**
- `Type2D` - No spatial processing
- `Type3DStationary` - Fixed world position
- `Type3DAttachedToObject` - Follows game object
- `TypeMusic` - Stereo music playback
- `TypeInEditor` - Editor preview mode

**Spatial Calculations:**
```cpp
// Update object position every frame
g_soundSystem->m_interface->GetObjectPosition(id, m_pos, m_vel);
g_soundLibrary3d->SetChannelPosition(m_channelIndex, m_pos, m_vel);
```
