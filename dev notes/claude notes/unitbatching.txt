# DEFCON UNIT BATCHING SYSTEM RESEARCH
# ====================================
# Research on how to safely batch all world objects while preserving rendering states

## CURRENT RENDERING ARCHITECTURE

### Base WorldObject System
- All units inherit from WorldObject base class
- WorldObject::Render() handles base sprite rendering + selection highlights
- WorldObject::RenderBatched() exists for sprite batching (BlitSprite)
- Each unit overrides Render() to add additional sprites on top

### Current Rendering Flow
1. MapRenderer::RenderObjects() loops through all world objects
2. Each object calls wobj->Render() individually 
3. Render() calls WorldObject::Render() for base sprite
4. Then renders additional sprites (aircraft counts, nuke indicators, etc.)

### Battleship Batching Success
- Battleships only render base sprite (no additional sprites)
- Successfully moved to RenderBatched() -> BlitSprite() 
- Gained 20 FPS because all battleships batched into single draw call
- Simple case: just base sprite, no complex state-dependent rendering

## BATCHING IMPLEMENTATION STRATEGIES

### STRATEGY 1: HYBRID BATCHING (RECOMMENDED)
**Concept**: Separate batching passes for different sprite types

```cpp
void MapRenderer::RenderObjects() {
    // Pass 1: Batch all base unit sprites
    for( WorldObject *wobj : allObjects ) {
        if( shouldBatchUnit(wobj) ) {
            wobj->RenderBaseBatched();  // Only base sprite
        }
    }
    g_renderer->FlushSpriteBuffer();  // Single draw call for all units
    
    // Pass 2: Batch all small aircraft sprites
    for( WorldObject *wobj : allObjects ) {
        if( wobj->HasAircraftSprites() ) {
            wobj->RenderAircraftBatched();
        }
    }
    g_renderer->FlushSpriteBuffer();  // Single draw call for all aircraft
    
    // Pass 3: Batch all nuke indicator sprites  
    for( WorldObject *wobj : allObjects ) {
        if( wobj->HasNukeSprites() ) {
            wobj->RenderNukesBatched();
        }
    }
    g_renderer->FlushSpriteBuffer();  // Single draw call for all nukes
    
    // Pass 4: Selection highlights (different blend mode)
    RenderSelectionHighlights();
}
```

**Advantages**:
- Preserves all existing rendering logic
- Achieves batching for each sprite type
- 3-4 draw calls instead of 100+ individual calls
- No risk of breaking unit states

### STRATEGY 2: MULTI-PASS RENDERING
**Concept**: Render units in multiple passes, batching compatible sprites

```cpp
// Phase 1: All base unit sprites (same shader/blend mode)
void RenderBaseSprites() {
    for( WorldObject *wobj : allObjects ) {
        wobj->RenderBaseSpriteBatched();
    }
    g_renderer->FlushSpriteBuffer();
}

// Phase 2: All aircraft indicators (same texture)
void RenderAircraftIndicators() {
    for( WorldObject *wobj : allObjects ) {
        if( wobj->m_type == TypeAirBase || wobj->m_type == TypeCarrier ) {
            wobj->RenderAircraftIndicatorsBatched();
        }
    }
    g_renderer->FlushSpriteBuffer();
}

// Phase 3: All nuke indicators (same texture)
void RenderNukeIndicators() {
    for( WorldObject *wobj : allObjects ) {
        if( wobj->m_type == TypeSilo || wobj->m_type == TypeSub ) {
            wobj->RenderNukeIndicatorsBatched();
        }
    }
    g_renderer->FlushSpriteBuffer();
}
```

### STRATEGY 3: SMART STATE CACHING (ADVANCED)
**Concept**: Pre-calculate sprite lists and batch render

```cpp
class UnitRenderCache {
    struct CachedSprite {
        Image* texture;
        float x, y, w, h;
        Colour color;
        int renderPass;  // 0=base, 1=aircraft, 2=nukes
    };
    
    LList<CachedSprite> m_cachedSprites[3];  // 3 render passes
    bool m_dirty;
    
    void UpdateCache(WorldObject* unit) {
        if( unit->m_lastUpdateTime > m_cacheTime ) {
            RebuildSpriteCache(unit);
            m_dirty = true;
        }
    }
    
    void RenderCachedSprites(int pass) {
        for( CachedSprite& sprite : m_cachedSprites[pass] ) {
            g_renderer->BlitSprite(sprite.texture, sprite.x, sprite.y, 
                                 sprite.w, sprite.h, sprite.color);
        }
        g_renderer->FlushSpriteBuffer();
    }
};
```

## RISK MITIGATION

### Critical Preservation Requirements
1. **State-dependent rendering**: Aircraft vs bomber sprites
2. **Dynamic counts**: Variable numbers of indicator sprites  
3. **Color changes**: Queued vs available indicators
4. **Position calculations**: Formation layouts for indicators
5. **Prediction**: Moving unit sprite positions
6. **Selection highlights**: Different blend modes

### Testing Strategy
1. **Unit Tests**: Each unit type rendering in all states
2. **State Transition Tests**: Verify sprite changes during state switches
3. **Performance Tests**: Frame rate monitoring
4. **Visual Tests**: Screenshot comparison system

### Rollback Plan
- Keep original `Render()` methods as fallback
- Runtime batching toggle: `g_renderer->SetBatchingEnabled(false)`
- Per-unit batching override for problematic units

## PERFORMANCE EXPECTATIONS

### Current Rendering (Worst Case)
- 100 units on screen
- Each unit: 1-10 sprite draws
- Total: 500-1000 draw calls per frame
- GPU bottleneck on sprite switching

### After Full Batching
- Base sprites: 1 draw call (all units)
- Aircraft indicators: 1 draw call (airbases/carriers)
- Nuke indicators: 1 draw call (silos/subs)  
- Selection highlights: 1 draw call
- Total: 4 draw calls per frame
- Expected FPS gain: 100-200% improvement

## CONCLUSION

The unit batching system is feasible but requires careful implementation to preserve the complex state-dependent rendering logic. The hybrid batching approach offers the best balance of performance gains while minimizing the risk of breaking existing functionality.

**Recommended Implementation Order**:
1. Base sprite batching (immediate 20+ FPS gain)
2. Aircraft indicator batching (airbase/carrier sprites)
3. Nuke indicator batching (silo/sub sprites)
4. Selection highlight batching
5. Advanced caching optimizations

Each phase should be thoroughly tested before proceeding to ensure no rendering states are corrupted during the transition to batched rendering.
