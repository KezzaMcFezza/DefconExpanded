================================================================================
BOMBER ORDER QUEUEING IMPLEMENTATION PLAN
(Thundy-style, adapted for DefconExpanded)
================================================================================

Scope: DefconExpanded has only 2 bomber states (0=air-to-sea, 1=nuke). No LACM.
Thundy has 3 states (0=standby, 1=nuke, 2=LACM/LASM). We adapt Thundy's nuke
queueing logic for DefconExpanded state 1 only.

--------------------------------------------------------------------------------
DESIGN CONSTRAINTS (user requirements)
--------------------------------------------------------------------------------

1. CARRIERS STAY PUT
   - Carriers must NEVER move towards launch targets, even when out of range.
   - Do NOT call AlterWaypoints/fleet move when carrier gets an attack order.
   - Carrier queues the order and stays put.

2. LAUNCH RANGE IS NOT A LAUNCH GATE (fuel indicator only)
   - For launching aircraft (fighters, bombers) from carriers and airbases,
     range is NOT a limiting factor for whether launch is permitted.
   - Range indicates fuel level for the aircraft - a distant target may mean
     the aircraft runs out of fuel before reaching it, but that's separate.
   - Carrier and AirBase MUST always accept and process launch orders, even
     when target is out of range. Launch immediately; aircraft flies toward
     target. (Bombers: queue order, launch bomber, bomber flies. Same for
     fighters.)

3. BOMBERS CAN QUEUE OUT-OF-RANGE TARGETS
   - Bombers can fly, so they MUST accept and queue attack orders even when
     the target is out of range. Bomber will fly towards target and fire
     when in range.
   - Bomber::RequestAction: accept regardless of range. SetWaypoint so bomber
     flies towards target.
   - Interface: allow left-click on enemy with bomber selected even when
     bomber is out of range.

4. CLICK SEMANTICS: RIGHT = WAYPOINT, LEFT = ATTACK (when applicable)
   - RIGHT CLICK: Movement/waypoint orders ONLY. Never adds attack orders.
     Units path to the location without attacking anything along the way.
   - LEFT CLICK:
     a) On valid combat target: HandleObjectAction (attack order)
     b) On EMPTY ground: If selection includes a NUKING unit (bomber in nuke
        mode, silo, sub), HandleObjectAction (attack order - nuke at coords).
        Nukes can be launched on empty space (no target object required).
     c) Else (empty + non-nuking, or friendly/invalid target): HandleSetWaypoint
   - This prevents bombers armed for nuke from auto-attacking when the player
     right-clicks to path them. Waypoint = pure movement.
   - File: source/renderer/map_renderer.cpp - left-click handler must branch
     on: underMouseId, valid combat target, and "has nuking unit" for empty.

5. NUKE ORDERS: STATIC TARGETS ONLY (never moving)
   - Nuke launch orders can only target: empty space, city, or building.
   - NEVER target moving units (ships, subs, fighters, bombers, etc.).
   - When user left-clicks on a MOVING target with nuking unit selected:
     Resolve to static: (a) building/city at those coordinates, or (b) -1 for
     empty space. Use the click (longitude, latitude) - find what's "underneath"
     the moving unit at that location.
   - ResolveNukeTarget(underMouseId, lon, lat): if underMouse->IsMovingObject(),
     then search for static at (lon,lat): city (exact or nearest), building
     (silo, radar, etc.). Return that objectId or -1. Never return moving ID.
   - Bomber::IsValidCombatTarget: when in nuke mode (state 1), remove the
     block that returns TargetTypeValid for moving sea objects. Nukes target
     static only. (Carrier/airbase bomber launch: same - they pass orders to
     bomber, which nukes static. Silo/sub: same.)

PHASE 3: BOMBER - ACTION (FIRE, AUTO-REPEAT, OUT-OF-RANGE RE-QUEUE)
--------------------------------------------------------------------------------

File: source/world/bomber.cpp
Function: Bomber::Action

CURRENT flow for state 1:
- In range: LaunchNuke, clear m_nukeTarget, m_bombingRun=true, fall through to
  MovingObject::Action (which decrements ammo)
- Out of range: SetNukeTarget, return (order is LOST)

TARGET (match Thundy):
- Add dead-target check at start of state 1 block:
  if ( targetObjectId != -1 ):
    WorldObject *targetObj = GetWorldObject( targetObjectId );
    if ( !targetObj || targetObj->m_life <= 0 ) return;  // don't consume ammo
- In range: WorldObject::Action first (decrements), then LaunchNuke
  (DefconExpanded calls MovingObject::Action which calls WorldObject::Action -
   keep that, but ensure order: decrement then launch)
- After launch: AUTO-REPEAT logic:
  remainingAfterQueue = m_states[1]->m_numTimesPermitted - m_actionQueue.Size();
  if ( remainingAfterQueue > 0 && m_actionQueue.Size() == 0 && targetObjectId != -1 ):
    create repeat ActionOrder, RequestAction(repeat), SetWaypoint
  else if ( m_actionQueue.Size() > 0 ):
    SetWaypoint to next queued target
- Out of range, targetObjectId == -1: SetNukeTarget, m_bombingRun=true, return
- Out of range, targetObjectId != -1: RE-QUEUE at front:
  ActionOrder *retry = new ActionOrder();
  retry->m_targetObjectId = targetObjectId;
  retry->m_longitude = longitude;
  retry->m_latitude = latitude;
  m_actionQueue.PutDataAtStart( retry );
  SetWaypoint( longitude, latitude );
  return
- State 1 block must return at end (do NOT fall through to MovingObject::Action
  for state 1 after handling - Thundy returns; we need MovingObject::Action only
  when we actually launched, for timer/ammo. Thundy calls WorldObject::Action
  directly. DefconExpanded uses MovingObject::Action. Ensure we call
  MovingObject::Action when we launch - before the auto-repeat block.)

NOTE: Ammo decrement - Thundy calls WorldObject::Action() explicitly before
Launch. DefconExpanded falls through to MovingObject::Action. Keep the
fall-through but ensure we only reach it when we launched. Currently when we
launch we don't return - we fall through. Good. When we don't launch (out of
range or dead target) we return. Good.

PHASE 4: BOMBER - UPDATE (FIRE FROM QUEUE + LEGACY m_nukeTarget)
--------------------------------------------------------------------------------

File: source/world/bomber.cpp
Function: Bomber::Update

CURRENT:
- Simple block: if state 1 and m_nukeTarget set and m_stateTimer==0 and in
  range, call Action(-1, m_nukeTargetLongitude, m_nukeTargetLatitude)
- No queue-based firing
- No "arrived but have queued attack" waypoint hold
- No hasQueuedAttack / hasAmmoAndWaypoint for landing decision
- No waypoint update to next queued target

TARGET (match Thundy):
1. After MoveToWaypoint(), add:
   if ( arrived && m_currentState == 1 && m_actionQueue.Size() > 0 ):
     SetWaypoint( next->m_longitude, next->m_latitude )
   (so we don't become "idle" and land before we can fire)

2. Replace the simple nuke firing block with unified block:
   if ( m_currentState == 1 && m_stateTimer == 0 ):
     targetLon, targetLat, targetObjId = from m_nukeTarget OR from m_actionQueue[0]
     firedFromLegacyTarget = (m_nukeTarget was set)
     hasTarget = m_nukeTarget set OR queue not empty
     if hasTarget and in range:
       if queue not empty: remove from queue, delete consumed
       Action( targetObjId, targetLon, targetLat )
       m_bombingRun = true
       if firedFromLegacyTarget:
         remainingAfterQueue = m_states[1]->m_numTimesPermitted - m_actionQueue.Size()
         if remainingAfterQueue > 0 && m_actionQueue.Size() == 0:
           create repeat with targetObjectId=-1, lon, lat
           RequestAction(repeat), SetWaypoint

3. Landing logic:
   hasQueuedAttack = (m_currentState==1) && m_actionQueue.Size() > 0
   hasAmmoAndWaypoint = (m_currentState==1) && m_states[1]->m_numTimesPermitted > 0
     && (m_targetLongitude != 0 || m_targetLatitude != 0)
   Land when: (m_bombingRun && ammo==0) OR (IsIdle() && !hasQueuedAttack && !hasAmmoAndWaypoint)

4. State 1 post-block:
   if ammo==0 && queue empty: SetState(0), Land
   else if queue not empty: update waypoint to next
   else if queue empty: ClearWaypoints, Land
