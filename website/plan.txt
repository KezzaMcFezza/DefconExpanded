=======================================================================
DEFCON REPLAY VIEWER - DYNAMIC FILE LOADING IMPLEMENTATION PLAN
=======================================================================

PURPOSE:
Transform the current hardcoded replay viewer into a dynamic system that can
load .dcrec files specified via command line arguments and served through
Node.js backend, enabling web-based replay viewing with flexible file selection.

CURRENT STATE ANALYSIS:
- Replay viewer mode works with REPLAY_VIEWER=1 flag
- Files are preloaded into WebAssembly virtual file system via CMake
- Hardcoded filename: "game_2024-10-14-20-44.dcrec" in RecordingSelectionWindow
- RecordingParser loads .dcrec files from virtual file system
- WebAssembly files currently in public folder
- Node.js server serves static files from publicDir

TARGET FUNCTIONALITY:
- Command line: ./defcon.html -l examplerecording.dcrec
- Command line: ./defcon.html -l demo_recordings/examplerecording.dcrec  
- Node.js backend launches replay viewer with specified file
- Dynamic file loading outside WebAssembly virtual file system
- WebAssembly files served from website root instead of public folder

STEP 3: IMPLEMENT DYNAMIC FILE LOADING FOR WEBASSEMBLY
=======================================================
Challenge: WebAssembly virtual file system vs external file access

Option A: HTTP Fetch Approach (RECOMMENDED)
- Create new function in RecordingParser to load from URL
- Use Emscripten's fetch API to download .dcrec from Node.js
- Parse downloaded data directly into memory
- No need to modify virtual file system

Option B: Virtual File System Population  
- Use Emscripten's FS.writeFile() to populate virtual FS at runtime
- Requires JavaScript integration

Implementation for Option A:
File: source/network/recordingparser.cpp
- Add constructor: RecordingParser(const std::string &url, Server *server)
- Add method: bool LoadFromURL(const std::string &url)
- Use emscripten_fetch_attr_t for HTTP requests

TECHNICAL CONSIDERATIONS:
=======================================================================

WEBASSEMBLY FILE LOADING CHALLENGES:
- WebAssembly can't directly access host file system
- Need HTTP requests to load external files
- CORS policies may affect file loading
- Large .dcrec files need streaming/chunked loading

SECURITY CONSIDERATIONS:
- Validate all file paths to prevent directory traversal
- Sanitize filenames in Node.js endpoints
- Implement proper CORS headers
- Rate limiting for file requests

PERFORMANCE OPTIMIZATION:
- Implement file caching in browser
- Use HTTP compression for .dcrec files
- Lazy loading for large recordings
- Progress indicators for file loading

BACKWARDS COMPATIBILITY:
- Keep existing hardcoded fallback for standalone builds
- Maintain current RecordingSelectionWindow interface
- Ensure normal (non-replay-viewer) builds unaffected
